RESUME LENGKAP PROGRESS PROJECT SMARTFARMING
============================================

Tujuan file ini:
File ini dibuat sebagai bahan belajar dan bahan diskusi lanjutan. Isinya menjelaskan apa saja yang sudah dikerjakan pada project SmartFarming, teknologi yang dipakai, logika alur sistem, status Kubernetes lokal, dan roadmap berikutnya.


1. RINGKASAN BESAR
------------------

Project ini adalah sistem SmartFarming untuk monitoring greenhouse. Aplikasi terdiri dari:

1. Frontend dashboard untuk menampilkan data sensor, grafik, progress panen, penggunaan aktuator, dan upload laporan.
2. farm-data-service untuk menerima data sensor IoT, menyimpan data ke PostgreSQL, dan menyediakan API metrik.
3. storage-service untuk upload, list, download, dan delete file laporan ke storage S3-compatible.
4. iot-simulator untuk membuat data sensor tiruan secara berkala.
5. PostgreSQL sebagai database telemetry dan konfigurasi tanaman.
6. MinIO sebagai pengganti Amazon S3 pada mode lokal.
7. Docker Compose untuk menjalankan stack lokal dengan mudah.
8. Kubernetes lokal/MicroK8s untuk latihan deployment ala EKS sebelum masuk AWS.

Status saat ini:
Fondasi lokal project sudah komplit untuk tahap local development dan Kubernetes lokal. Service utama sudah bisa berjalan dan saling terhubung. Bagian cloud AWS seperti EKS asli, ECR, RDS, S3, IRSA, Terraform/CDK, backup cloud, budget alert, dan cost tagging masih menjadi tahap berikutnya.


2. TARGET AKHIR BERDASARKAN DESKRIPSI PROJECT
---------------------------------------------

Target akhir project adalah menjalankan sistem SmartFarming di AWS dengan arsitektur yang lebih production-like:

1. Amazon EKS sebagai Kubernetes cluster.
2. Amazon ECR sebagai container image registry.
3. Amazon RDS PostgreSQL sebagai database managed.
4. Amazon S3 sebagai object storage untuk laporan.
5. IAM Roles for Service Accounts atau IRSA agar Pod di EKS bisa akses AWS service tanpa menyimpan access key di container.
6. Terraform atau AWS CDK untuk membuat infrastruktur secara otomatis.
7. Backup dan restore untuk database dan object storage.
8. Cost tagging, budget, dan monitoring biaya.
9. Kubernetes manifest yang rapi dengan namespace, service account, resource request/limit, probe, dan autoscaling.

Yang sudah kita kerjakan sekarang adalah versi lokal dan Kubernetes lokal sebagai pondasi sebelum migrasi ke AWS.


3. TEKNOLOGI YANG DIPAKAI
-------------------------

Bahasa dan framework aplikasi:

1. Python 3.11
   Dipakai oleh farm-data-service, storage-service, dan iot-simulator.

2. FastAPI
   Dipakai untuk membuat REST API pada farm-data-service dan storage-service.

3. Uvicorn
   ASGI server untuk menjalankan aplikasi FastAPI.

4. SQLAlchemy
   ORM untuk membuat model database dan query ke PostgreSQL pada farm-data-service.

5. Pydantic
   Validasi request body, terutama data telemetry sensor.

6. boto3
   SDK AWS untuk koneksi ke S3-compatible storage. Pada lokal, boto3 diarahkan ke MinIO.

7. requests
   Dipakai iot-simulator untuk mengirim HTTP POST telemetry ke farm-data-service.

Frontend:

1. HTML, CSS, JavaScript vanilla
   Dipakai untuk dashboard web.

2. Chart.js
   Dipakai untuk menampilkan grafik harian suhu, kelembaban, cahaya, dan energi.

3. Nginx
   Dipakai untuk menyajikan file static frontend dan reverse proxy ke backend.

Storage dan database:

1. PostgreSQL 16 Alpine
   Database lokal untuk menyimpan sensor_data dan crop_config.

2. MinIO
   Storage lokal yang kompatibel dengan S3. Ini menggantikan Amazon S3 saat development lokal.

Container dan orkestrasi:

1. Docker
   Untuk build image tiap service.

2. Docker Compose
   Untuk menjalankan semua service lokal dalam satu command.

3. Kubernetes/MicroK8s
   Untuk menjalankan stack dalam cluster lokal.

4. Kubernetes ConfigMap
   Untuk konfigurasi non-rahasia seperti interval simulator, endpoint service, dan parameter crop.

5. Kubernetes Secret
   Untuk konfigurasi sensitif seperti DATABASE_URL, password PostgreSQL, dan credential MinIO lokal.

6. Kubernetes Deployment
   Untuk menjalankan Pod aplikasi dengan replica.

7. Kubernetes Service ClusterIP
   Untuk komunikasi antar service di dalam cluster.

8. Kubernetes PersistentVolumeClaim
   Untuk menyimpan data PostgreSQL dan MinIO agar tidak hilang ketika Pod restart.

9. Kubernetes HorizontalPodAutoscaler atau HPA
   Untuk autoscaling berdasarkan CPU.

10. Kubernetes probes
    Startup probe, readiness probe, dan liveness probe dipakai agar Kubernetes tahu kapan container siap, sehat, atau perlu restart.

5. ALUR LOGIKA SISTEM
---------------------

Alur data telemetry:

1. iot-simulator membuat data sensor tiruan.
2. Data sensor berisi:
   - temperature_c
   - humidity_percent
   - light_lux
   - energy_kwh
3. iot-simulator mengirim data ke farm-data-service melalui endpoint POST /telemetry.
4. farm-data-service memvalidasi data dengan Pydantic.
5. farm-data-service menyimpan data ke tabel sensor_data di PostgreSQL.
6. Frontend mengambil data terbaru dari GET /api/metrics/latest.
7. Nginx frontend meneruskan /api/... ke farm-data-service.
8. Frontend menampilkan data pada card dashboard.
9. Frontend mengambil data harian dari GET /api/metrics/daily.
10. Chart.js menggambar grafik harian suhu, kelembaban, cahaya, dan energi.

Alur progress panen:

1. farm-data-service menyimpan konfigurasi crop pada tabel crop_config.
2. Saat service start, jika crop_config belum ada, service membuat default config.
3. Default config diambil dari environment:
   - CROP_DAYS_SINCE_PLANTING
   - CROP_EXPECTED_HARVEST_DAYS
   - CROP_TARGET_ENERGY_KWH_PER_DAY
4. Endpoint GET /harvest-progress menghitung:
   - tanggal tanam
   - total hari target panen
   - hari yang sudah berjalan
   - persentase progress
   - status growing atau harvest_ready
5. Frontend menampilkan progress bar menuju panen.

Alur konsumsi energi:

1. farm-data-service mengambil semua data sensor hari ini.
2. Service menghitung total energy_kwh hari ini.
3. Total energi dibandingkan dengan target harian dari crop_config.
4. Endpoint GET /energy-consumption mengembalikan:
   - today_energy_kwh
   - target_energy_kwh_per_day
   - percentage_of_target
   - status ok atau exceeded

Alur file/laporan:

1. User memilih file laporan di frontend.
2. Frontend mengirim file ke POST /storage/upload.
3. Nginx frontend meneruskan /storage/... ke storage-service.
4. storage-service menerima file dengan FastAPI UploadFile.
5. storage-service memakai boto3 untuk upload file ke bucket reports.
6. Pada lokal, bucket reports ada di MinIO.
7. Frontend mengambil daftar file dari GET /storage/files.
8. User bisa download file melalui GET /storage/files/{filename}.
9. User bisa hapus file melalui DELETE /storage/files/{filename}.


6. DETAIL farm-data-service
---------------------------

Lokasi utama:
farm-data-service/app.py

Fungsi utama:

1. Membuat koneksi ke PostgreSQL melalui DATABASE_URL.
2. Membuat tabel sensor_data.
3. Membuat tabel crop_config.
4. Menyimpan data telemetry sensor.
5. Mengambil data sensor terbaru.
6. Mengambil data sensor harian.
7. Menghitung progress panen.
8. Menghitung konsumsi energi.
9. Menyediakan endpoint health check.

Model database:

1. SensorData
   Tabel: sensor_data
   Kolom:
   - id
   - timestamp
   - temperature_c
   - humidity_percent
   - light_lux
   - energy_kwh

2. CropConfig
   Tabel: crop_config
   Kolom:
   - id
   - planting_date
   - expected_harvest_days
   - target_energy_kwh_per_day

Endpoint penting:

1. POST /telemetry
   Menerima data sensor dari simulator dan menyimpan ke database.

2. GET /metrics/latest
   Mengambil data sensor terbaru.

3. GET /metrics/daily
   Mengambil seluruh data sensor pada tanggal tertentu atau hari ini.

4. GET /harvest-progress
   Menghitung progress menuju panen.

5. GET /energy-consumption
   Menghitung total energi hari ini dibanding target.

6. GET /health
   Mengembalikan status service.

Logika startup:

1. FastAPI memakai lifespan.
2. Saat aplikasi start, init_database() dijalankan.
3. init_database() mencoba membuat tabel dengan SQLAlchemy.
4. Jika database belum siap, service retry beberapa kali sesuai DB_INIT_RETRIES dan DB_INIT_DELAY_SECONDS.
5. Setelah tabel siap, seed_crop_config() membuat konfigurasi crop default jika belum ada data.

Kenapa retry diperlukan:
Di container/Kubernetes, PostgreSQL bisa saja belum siap saat farm-data-service mulai berjalan. Retry membuat service lebih tahan terhadap urutan startup yang tidak selalu pasti.


7. DETAIL storage-service
-------------------------

Lokasi utama:
storage-service/app.py

Fungsi utama:

1. Membuat koneksi boto3 ke S3-compatible endpoint.
2. Pada lokal, endpoint diarahkan ke MinIO.
3. Memastikan bucket reports tersedia.
4. Upload file.
5. List file.
6. Download file.
7. Delete file.
8. Health check.

Endpoint penting:

1. POST /upload
   Upload file ke bucket.

2. GET /files
   Melihat daftar file dalam bucket.

3. GET /files/{filename}
   Download file tertentu.

4. DELETE /files/{filename}
   Menghapus file tertentu dari bucket.

5. GET /health
   Mengecek service.

Logika startup:

1. FastAPI menjalankan ensure_bucket() saat startup.
2. ensure_bucket() mencoba head_bucket.
3. Jika bucket belum ada, service membuat bucket.
4. Jika MinIO belum siap, service retry berdasarkan S3_INIT_RETRIES dan S3_INIT_DELAY_SECONDS.

Catatan cloud:
Pada AWS nanti, storage-service sebaiknya tidak memakai S3_ACCESS_KEY dan S3_SECRET_KEY statis. Akses ke S3 sebaiknya memakai IRSA agar Pod mendapat permission dari IAM Role.


8. DETAIL iot-simulator
-----------------------

Lokasi utama:
iot-simulator/simulator.py

Fungsi utama:

1. Membuat data sensor tiruan.
2. Mengirim data ke farm-data-service setiap INTERVAL_SECONDS.
3. Endpoint target diambil dari FARM_DATA_URL.

Sensor yang disimulasikan:

1. temperature_c
   Suhu greenhouse.

2. humidity_percent
   Kelembaban udara.

3. light_lux
   Intensitas cahaya.

4. energy_kwh
   Konsumsi energi.

Logika data sensor:

1. Nilai awal dibuat secara random dalam rentang yang masuk akal.
2. Setiap siklus, nilai berubah sedikit demi sedikit.
3. Fungsi clamp() menjaga nilai tetap dalam batas minimum dan maksimum.
4. Pola ini membuat data terlihat lebih natural daripada random total setiap menit.

Interval lokal:
Simulator mengirim telemetry setiap 60 detik.

1. FARM_DATA_API = "/api"
2. STORAGE_API = "/storage"

Browser user tidak bisa resolve DNS internal Kubernetes seperti farm-data-service.farm-db-svc.svc.cluster.local. Karena itu request browser diarahkan ke domain frontend dulu, lalu Nginx di dalam cluster meneruskan ke service internal.


10. DETAIL NGINX FRONTEND
-------------------------

Ada dua konfigurasi:

1. frontend-app/nginx.local.conf
   Dipakai untuk Docker Compose lokal.
   Upstream:
   - farm-data-service:8000
   - storage-service:8000

2. frontend-app/nginx.conf
   Dipakai untuk Kubernetes lokal.
   Upstream:
   - farm-data-service.farm-db-svc.svc.cluster.local:8000
   - storage-service.storage-svc.svc.cluster.local:8000

Perubahan penting pada nginx.conf Kubernetes:

1. Menambahkan resolver MicroK8s:
   resolver 10.152.183.10 valid=10s ipv6=off;

2. Menyimpan upstream dalam variable:
   - farm_data_upstream
   - storage_upstream

3. Menambahkan rewrite:
   - /api/health menjadi /health sebelum dikirim ke farm-data-service
   - /storage/health menjadi /health sebelum dikirim ke storage-service

Masalah yang diselesaikan:

1. Nginx dapat gagal start jika DNS service Kubernetes belum resolve pada saat container start.
2. Dengan resolver dan variable upstream, Nginx bisa resolve DNS saat runtime.
3. Rewrite path mencegah backend menerima path yang salah seperti /api/metrics/latest, karena backend sebenarnya punya endpoint /metrics/latest.

16. MASALAH KUBERNETES YANG SUDAH DISELESAIKAN
----------------------------------------------

Masalah:
Cluster MicroK8s sempat punya node lama bernama staging dengan status NotReady. Node aktif yang benar adalah app-server.

Dampak:

1. Beberapa pod atau resource lama terkait registry/storage bisa tersangkut ke node lama.
2. Registry localhost:32000 sempat tidak bisa dipakai normal.
3. Ada indikasi masalah jaringan/Calico akibat IP conflict dari node lama.

Yang dilakukan:

1. Menghapus stale node staging dari cluster.
2. Restart Calico daemonset.
3. Memastikan node aktif app-server kembali Ready.
4. Menggunakan fallback import image langsung ke MicroK8s containerd.

Status setelah diperbaiki:

1. Node aktif: app-server.
2. Node status: Ready.
3. Internal IP aktif yang pernah diverifikasi: 100.87.119.107.
4. Service aplikasi berjalan dengan pod ready.

19. CATATAN RESOURCE LAMA/STAGING
---------------------------------

Masih ada kemungkinan resource lama dari percobaan sebelumnya, misalnya deployment/service MinIO lama atau PVC lama. Kita sengaja belum menghapus semua resource lama secara agresif karena:

1. Menghapus PVC bisa menghapus data.
2. Perlu dibedakan mana resource aktif dan mana resource sisa.
3. Cleanup sebaiknya dilakukan terkontrol.

Rekomendasi cleanup berikutnya:

1. List semua resource per namespace.
2. Identifikasi resource aktif:
   - postgres-local
   - minio-local
   - farm-data-service
   - storage-service
   - frontend
   - iot-simulator
3. Hapus deployment/service lama yang tidak dipakai.
4. Jangan hapus PVC sebelum yakin data tidak diperlukan.

22. ROADMAP TAHAP SELANJUTNYA
-----------------------------

Tahap 1: Cleanup Kubernetes lokal

Tujuan:
Membersihkan resource lama agar cluster lokal tidak membingungkan.

Pekerjaan:

1. List semua deployment, service, pod, pvc, hpa, configmap, secret.
2. Tandai resource aktif dan resource sisa.
3. Hapus deployment/service lama yang tidak dipakai.
4. Hapus PVC hanya jika sudah yakin data boleh hilang.
5. Update KUBERNETES_LOCAL.md jika ada perubahan command.

Tahap 2: Rapikan image dan versioning

Tujuan:
Membuat image lebih siap untuk cloud.

Pekerjaan:

1. Buat tag versioned, misalnya v0.1.0.
2. Hindari hanya bergantung pada latest.
3. Pastikan Dockerfile semua service aman dan konsisten.
4. Pertimbangkan user non-root untuk iot-simulator juga.
5. Tambahkan .dockerignore jika belum ada.

Tahap 3: Siapkan ECR

Tujuan:
Menaruh image di registry AWS.

Pekerjaan:

1. Buat repository ECR:
   - farm-data-service
   - storage-service
   - frontend
   - iot-simulator
2. Login Docker ke ECR.
3. Tag image lokal menjadi image ECR.
4. Push image ke ECR.
5. Update manifest Kubernetes agar image mengarah ke ECR.

Tahap 4: Infrastructure as Code

Tujuan:
Membuat infrastruktur AWS secara repeatable.

Pekerjaan:

1. Buat Terraform/CDK untuk VPC.
2. Buat Terraform/CDK untuk EKS.
3. Buat Terraform/CDK untuk ECR.
4. Buat Terraform/CDK untuk RDS PostgreSQL.
5. Buat Terraform/CDK untuk S3 bucket.
6. Buat IAM Role dan policy.
7. Buat OIDC provider untuk EKS.

Tahap 5: IRSA

Tujuan:
Mengganti static credential dengan IAM Role.

Pekerjaan:

1. Buat IAM policy untuk akses S3.
2. Buat IAM Role yang trust ke ServiceAccount Kubernetes.
3. Tambahkan annotation eks.amazonaws.com/role-arn pada ServiceAccount.
4. Hapus S3_ACCESS_KEY dan S3_SECRET_KEY dari Secret cloud.
5. Uji storage-service upload/list/download/delete ke S3 AWS.

Tahap 6: Migrasi database ke RDS

Tujuan:
Mengganti PostgreSQL lokal di cluster dengan RDS managed database.

Pekerjaan:

1. Buat RDS PostgreSQL.
2. Atur security group agar EKS bisa konek ke RDS.
3. Update DATABASE_URL ke endpoint RDS.
4. Simpan password di Secret atau AWS Secrets Manager.
5. Uji farm-data-service startup, create table, telemetry, dan query metric.

Tahap 7: Deploy ke EKS

Tujuan:
Menjalankan aplikasi di Kubernetes AWS.

Pekerjaan:

1. Deploy namespace, ServiceAccount, ConfigMap, dan Secret.
2. Deploy farm-data-service, storage-service, frontend, dan iot-simulator.
3. Gunakan image dari ECR.
4. Buat Ingress/LoadBalancer untuk frontend.
5. Uji akses dashboard dari browser.
6. Uji semua API.

Tahap 8: Backup, restore, dan cost control

Tujuan:
Melengkapi syarat operasional cloud.

Pekerjaan:

1. Aktifkan backup RDS.
2. Buat panduan restore RDS.
3. Aktifkan versioning atau lifecycle policy S3 jika diperlukan.
4. Tambahkan tags pada resource AWS.
5. Buat AWS Budget alert.
6. Dokumentasikan estimasi biaya.

Tahap 9: CI/CD

Tujuan:
Membuat proses build dan deploy lebih otomatis.

Pekerjaan:

1. Buat pipeline build image.
2. Push image ke ECR.
3. Jalankan test minimal.
4. Deploy ke EKS.
5. Gunakan environment staging/dev.

Tahap 10: Finalisasi dokumentasi dan presentasi

Tujuan:
Membuat project siap dikumpulkan/dipresentasikan.

Pekerjaan:

1. Update README atau docs utama.
2. Update LOCAL_RUN.md.
3. Update KUBERNETES_LOCAL.md.
4. Buat arsitektur diagram.
5. Buat script demo.
6. Buat checklist testing.
7. Buat catatan known limitation.
