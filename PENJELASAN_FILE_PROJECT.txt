PENJELASAN FILE PROJECT SMARTFARMING
====================================

Catatan:
- File smartfarming.zip tidak dibahas, sesuai arahan.
- File __pycache__ adalah file otomatis dari Python. File ini bukan bagian desain project dan tidak perlu dipelajari.
- Project ini masih tahap lokal. Jadi beberapa hal pada PDF yang sifatnya AWS cloud, seperti EKS, IRSA, RDS, S3 asli, Terraform/CDK, budget, dan backup cloud belum sepenuhnya ada di direktori aktif ini.


1. FILE DOKUMEN UTAMA
---------------------

1.1 final-project-cloud-admin-smartfarming.pdf
Tujuan:
File ini adalah draft/deskripsi project. Isinya menjelaskan target akhir project SmartFarming:
- Frontend untuk visualisasi data IoT.
- Backend farm-data-service untuk menyimpan data sensor ke PostgreSQL/RDS.
- Backend storage-service untuk menyimpan file ke S3.
- IoT simulator untuk membuat data sensor tiruan.
- Target cloud memakai Amazon EKS, IRSA, RDS PostgreSQL, S3, ECR, dan IaC dengan Terraform/CDK.

Fungsi dalam project:
PDF ini menjadi acuan utama untuk mengecek apakah isi folder sudah sesuai rencana.

Status saat ini:
Direktori lokal sudah mengarah ke konsep PDF, tetapi masih lebih fokus ke local development memakai Docker Compose, PostgreSQL lokal, dan MinIO sebagai pengganti S3 lokal.


1.2 LOCAL_RUN.md
Tujuan:
File panduan untuk menjalankan project secara lokal.

Isi penting:
- Perintah menjalankan semua service:
  docker compose -f docker-compose.local.yml up --build
- Daftar endpoint lokal:
  - Dashboard: http://localhost:8080
  - Farm Data API docs: http://localhost:8000/docs
  - Storage API docs: http://localhost:8001/docs
  - MinIO console: http://localhost:9001
- Cara menghentikan stack.
- Cara reset data lokal.

Fungsi dalam project:
Ini adalah file pertama yang sebaiknya dibaca saat ingin menjalankan project di laptop/local machine.


2. FILE ORKESTRASI LOKAL
------------------------

2.1 docker-compose.local.yml
Tujuan:
Menjalankan seluruh sistem SmartFarming di lokal dalam beberapa container.

Service yang didefinisikan:
- postgres
  Database lokal untuk menyimpan data sensor farming.

- minio
  Storage lokal yang kompatibel dengan S3. Ini dipakai sebagai pengganti Amazon S3 saat tahap lokal.

- farm-data-service
  Backend API untuk menerima telemetry sensor, membaca metric terbaru, membaca data harian, menghitung progress panen, dan menghitung konsumsi energi.

- storage-service
  Backend API untuk upload, list, download, dan delete file laporan.

- frontend
  Dashboard web statis yang diakses dari browser.

- iot-simulator
  Service yang mengirim data sensor otomatis ke farm-data-service setiap 60 detik.

Fungsi dalam project:
File ini adalah pusat local development. Dengan satu command Docker Compose, semua komponen dapat hidup bersama.

Hubungan dengan PDF:
Sesuai dengan tips PDF untuk menguji frontend, backend, dan PostgreSQL secara lokal sebelum pindah ke cloud.


3. DIREKTORI farm-data-service
------------------------------

Direktori ini berisi backend utama untuk data farming dan telemetry IoT.

3.1 farm-data-service/app.py
Tujuan:
Kode aplikasi FastAPI untuk mengelola data sensor dan data tanaman.

Fungsi utama:
- Membuat koneksi ke PostgreSQL melalui DATABASE_URL.
- Membuat tabel sensor_data.
- Membuat tabel crop_config.
- Menyimpan data telemetry dari IoT simulator.
- Mengambil metric sensor terbaru.
- Mengambil data metric harian untuk grafik frontend.
- Menghitung progress menuju panen.
- Menghitung konsumsi energi harian.
- Menyediakan endpoint health check.

Endpoint penting:
- POST /telemetry
  Menerima data sensor dari iot-simulator.

- GET /metrics/latest
  Mengambil data sensor terbaru untuk card dashboard.

- GET /metrics/daily
  Mengambil data sensor harian untuk grafik.

- GET /harvest-progress
  Menghitung progress tanaman berdasarkan tanggal tanam dan target hari panen.

- GET /energy-consumption
  Menghitung total energi hari ini dibanding target.

- GET /health
  Mengecek apakah service hidup.

Hubungan dengan PDF:
Ini mewakili farm-data-service yang pada target cloud nanti seharusnya terhubung ke Amazon RDS PostgreSQL.

Status lokal:
Sudah cocok untuk tahap lokal karena memakai PostgreSQL container dari Docker Compose.


3.2 farm-data-service/requirements.txt
Tujuan:
Daftar dependency Python untuk farm-data-service.

Isi penting:
- fastapi
- uvicorn
- psycopg2-binary
- sqlalchemy
- python-dotenv
- pydantic

Fungsi dalam project:
Dipakai oleh Dockerfile saat build image backend.


3.3 farm-data-service/Dockerfile
Tujuan:
Membuat container image untuk farm-data-service.

Yang dilakukan:
- Memakai base image python:3.11-slim.
- Membuat user non-root appuser.
- Menginstall dependency dari requirements.txt.
- Menyalin app.py.
- Menjalankan aplikasi dengan uvicorn.

Hubungan dengan PDF:
Sesuai dengan syarat PDF bahwa setiap service punya Dockerfile, memakai base image kecil, dan backend berjalan sebagai non-root user.


3.4 farm-data-service/farmdata-deploy.yaml
Tujuan:
Manifest Kubernetes untuk menjalankan farm-data-service.

Isi manifest:
- Deployment farm-data-service.
- Service ClusterIP.
- HorizontalPodAutoscaler.

Fungsi:
Dipakai saat ingin menjalankan service ini di Kubernetes atau MicroK8s.

Status terhadap PDF:
Yang sudah sesuai:
- Ada Deployment.
- Ada ClusterIP Service.
- Ada 2 replicas.
- Ada CPU/memory requests dan limits.
- Ada HPA.

Yang belum sesuai penuh:
- Belum ada ServiceAccount khusus.
- Belum ada serviceAccountName pada pod.
- Belum ada anotasi IRSA.
- DATABASE_URL masih memuat password langsung, cocok untuk lokal tetapi belum cocok untuk cloud production.


4. DIREKTORI storage-service
----------------------------

Direktori ini berisi backend untuk pengelolaan file/laporan.

4.1 storage-service/app.py
Tujuan:
Kode aplikasi FastAPI untuk operasi file ke storage S3-compatible.

Fungsi utama:
- Koneksi ke storage menggunakan boto3.
- Membuat bucket jika belum ada.
- Upload file.
- List file.
- Download file.
- Delete file.
- Health check.

Endpoint penting:
- POST /upload
  Upload file laporan.

- GET /files
  Melihat daftar file.

- GET /files/{filename}
  Download file tertentu.

- DELETE /files/{filename}
  Menghapus file tertentu.

- GET /health
  Mengecek service.

Hubungan dengan PDF:
Ini mewakili storage-service yang pada target cloud nanti seharusnya memakai Amazon S3.

Status lokal:
Sudah cocok untuk tahap lokal karena memakai MinIO sebagai pengganti S3.

Catatan penting:
Di lokal, service memakai S3_ACCESS_KEY dan S3_SECRET_KEY untuk koneksi ke MinIO. Ini normal untuk lokal, tetapi pada AWS EKS target akhirnya harus diganti dengan IRSA agar tidak menyimpan static AWS key.

Isi manifest:

Status terhadap PDF:

Yang belum sesuai penuh:

5. DIREKTORI iot-simulator
--------------------------

Direktori ini berisi service pembuat data sensor tiruan.

5.1 iot-simulator/simulator.py
Tujuan:
Mengirim data sensor otomatis ke farm-data-service.

Data yang dibuat:
- temperature_c
- humidity_percent
- light_lux
- energy_kwh

Cara kerja:
- Membuat nilai awal sensor secara random.
- Mengubah nilai sedikit demi sedikit agar terlihat seperti data sensor real.
- Mengirim data ke endpoint /telemetry di farm-data-service.
- Mengulang setiap INTERVAL_SECONDS, default 60 detik.

Hubungan dengan PDF:
PDF menyebut ada IoT sensors and actuators simulator. File ini adalah implementasi lokalnya.


5.2 iot-simulator/Dockerfile
Tujuan:
Membuat container image untuk simulator.

Yang dilakukan:
- Memakai python:3.11-slim.
- Menyalin simulator.py.
- Menginstall requests.
- Menjalankan simulator.py.

Catatan:
Dockerfile ini belum membuat user non-root seperti dua backend lain. Untuk lokal masih bisa jalan, tetapi jika ingin lebih rapi sesuai standar PDF, sebaiknya dibuat non-root juga.


5.3 iot-simulator/simulator-deploy.yaml
Tujuan:
Manifest Kubernetes untuk menjalankan iot-simulator.

Isi:
- Deployment iot-simulator.
- Environment FARM_DATA_URL.
- Environment INTERVAL_SECONDS.
- CPU/memory requests dan limits.

Status:
Cukup untuk simulator lokal/Kubernetes lokal, tetapi belum ada ServiceAccount. Karena simulator hanya mengirim data ke backend internal, IRSA mungkin tidak wajib untuk simulator.

8. ALUR KERJA SISTEM LOKAL
--------------------------

Alur data utama:
1. iot-simulator membuat data sensor.
2. iot-simulator mengirim data ke farm-data-service melalui POST /telemetry.
3. farm-data-service menyimpan data ke PostgreSQL.
4. frontend mengambil data terbaru dan data harian dari farm-data-service.
5. frontend menampilkan card metric dan grafik.
6. user bisa upload laporan dari frontend.
7. frontend mengirim file ke storage-service.
8. storage-service menyimpan file ke MinIO.

Alur akses browser:
1. User membuka http://localhost:8080.
2. Nginx frontend melayani index.html, style.css, app.js, dan asset logo.
3. Jika frontend memanggil /api, Nginx meneruskan request ke farm-data-service.
4. Jika frontend memanggil /storage, Nginx meneruskan request ke storage-service.
